Patrón AAA

Arrange · Act · Assert

# lab_01/services/calculadora.py

def calcular_precio_final(precio: float, descuento: float) -> float:
    if not 0 <= descuento <= 100:
        raise ValueError("Descuento fuera de rango")
    return precio * (1 - descuento / 100)
# lab_01/tests/test_calculadora.py

def test_descuento_valido():
    # Arrange
    precio    = 100.0
    descuento = 20

    # Act
    resultado = calcular_precio_final(precio, descuento)

    # Assert
    assert resultado == 80.0

def test_descuento_invalido_lanza_error():
    import pytest
    with pytest.raises(ValueError):
        calcular_precio_final(100, 150)

pytest

Fixtures: estado reutilizable

# conftest.py — disponible en todo el proyecto

import pytest

@pytest.fixture
def usuario_base():
    return {
        "id": 1,
        "nombre": "Ana García",
        "email": "ana@ejemplo.com",
        "activo": True,
    }

@pytest.fixture
def db_en_memoria():
    # Se ejecuta antes de cada test
    db = crear_db_temporal()
    db.poblar_datos_base()
    yield db         # ← el test recibe esto
    db.limpiar()     # teardown automático

# Usando la fixture
def test_usuario_activo(usuario_base):
    assert usuario_base["activo"] is True

Las fixtures evitan el código duplicado (DRY) y garantizan que cada test comience con un estado conocido.

Aislamiento

Mocking: reemplazar dependencias

from unittest.mock import patch, MagicMock
import pytest

# Servicio que depende de un cliente externo
class NotificadorEmail:
    def enviar(self, destinatario, mensaje):
        # ...llama a SendGrid, AWS SES, etc.
        pass

class ServicioRegistro:
    def __init__(self, notificador: NotificadorEmail):
        self.notificador = notificador

    def registrar_usuario(self, email):
        # lógica de negocio...
        self.notificador.enviar(email, "Bienvenido")
        return {"ok": True}

# ✅ Test SIN llamar al email real
def test_registrar_envia_email():
    mock_notif = MagicMock()
    servicio   = ServicioRegistro(mock_notif)

    resultado  = servicio.registrar_usuario("x@x.com")

    mock_notif.enviar.assert_called_once_with(
        "x@x.com", "Bienvenido"
    )
    assert resultado["ok"] is True

¿Cuándo mockear?

  • Servicios de email / SMS
  • APIs externas (Stripe, Google)
  • Base de datos (en pruebas unitarias)
  • Fecha y hora actual (datetime.now())
  • Generación de IDs aleatorios

¿Cuándo NO mockear?

  • Tu propia lógica de negocio
  • En pruebas de integración (usa la DB real)
  • Cuando el mock se vuelve más complejo que la dependencia

¿Qué mide?

Qué porcentaje de líneas del código de producción fue ejecutado durante las pruebas.

Objetivo razonable

Entre 70% y 85%. El 100% no garantiza calidad, solo que cada línea se ejecutó alguna vez.

Trampa común

Cobertura alta con asserts débiles. Una prueba puede ejecutar código sin verificar nada útil.

FastAPI

TestClient: probar la API completa

# app/main.py
from fastapi import FastAPI
from pydantic import BaseModel

app = FastAPI()
items_db = {}

class Item(BaseModel):
    nombre: str
    precio: float

@app.post("/items/", status_code=201)
def crear_item(item: Item):
    items_db[item.nombre] = item
    return item

@app.get("/items/{nombre}")
def obtener_item(nombre: str):
    if nombre not in items_db:
        raise HTTPException(404)
    return items_db[nombre]
# tests/test_items.py
from fastapi.testclient import TestClient
from app.main import app

client = TestClient(app)

def test_crear_item():
    resp = client.post("/items/", json={
        "nombre": "Mango",
        "precio": 2.5
    })
    assert resp.status_code == 201
    assert resp.json()["nombre"] == "Mango"

def test_item_no_existe():
    resp = client.get("/items/inexistente")
    assert resp.status_code == 404

def test_crear_y_obtener():
    client.post("/items/", json={
        "nombre": "Papaya", "precio": 3.0
    })
    resp = client.get("/items/Papaya")
    assert resp.json()["precio"] == 3.0
Integración con BD

Base de datos de prueba con SQLite

# conftest.py
import pytest
from sqlalchemy import create_engine
from sqlalchemy.orm import sessionmaker
from fastapi.testclient import TestClient
from app.main import app, get_db
from app.database import Base

# BD temporal en memoria
TEST_URL = "sqlite:///./test.db"
engine   = create_engine(TEST_URL)
Session  = sessionmaker(bind=engine)

@pytest.fixture(scope="function")
def db_test():
    Base.metadata.create_all(engine)
    session = Session()
    yield session
    session.close()
    Base.metadata.drop_all(engine)

@pytest.fixture
def client_test(db_test):
    def override_get_db():
        yield db_test
    app.dependency_overrides[get_db] = override_get_db
    with TestClient(app) as c:
        yield c
    app.dependency_overrides.clear()

Flujo de prueba con BD

1. Crear BD SQLite temporal
↓
2. Inyectar con dependency_overrides
↓
3. Ejecutar el test
↓
4. Limpiar con drop_all

¿Por qué SQLite? Es liviano, sin servidor, y compatible con SQLAlchemy. Para producción usarás PostgreSQL, pero las pruebas deben ser rápidas y aisladas.

# tests/e2e/test_flujo_items.py
import pytest
from playwright.sync_api import Page, expect

BASE = "http://localhost:8000"

@pytest.mark.e2e
def test_crear_item_desde_ui(page: Page):
    # Navegar al formulario
    page.goto(f"{BASE}/static/index.html")

    # Llenar el formulario
    page.fill("#nombre", "Mango")
    page.fill("#precio", "2.5")
    page.click("#btn-guardar")

    # Verificar resultado
    expect(page.locator("#mensaje")).to_contain_text(
        "Guardado exitosamente"
    )

@pytest.mark.e2e
def test_item_aparece_en_lista(page: Page):
    page.goto(f"{BASE}/static/index.html")
    item = page.locator(".item-card").first()
    expect(item).to_be_visible()

Foco del lab

Eficiencia de Rendimiento

La capacidad del sistema de responder en tiempos aceptables bajo carga de trabajo real o extrema.

Sub-característica¿Qué mide?
Tiempo de respuestaLatencia por petición
ThroughputPeticiones por segundo (RPS)
Utilización de recursosCPU, RAM, conexiones de BD

ISO 25010 define qué medir. K6 es la herramienta que ejecuta esa medición contra tu API FastAPI.

Rendimiento

Tipos de pruebas de rendimiento

📈

Carga (Load Test)

Simula el volumen de usuarios esperado en condiciones normales. ¿Rinde bien en el día a día?

Ej: 100 usuarios simultáneos durante 5 min

💥

Estrés (Stress Test)

Supera la capacidad máxima para encontrar el punto de quiebre del sistema.

Ej: incrementar hasta 1000 VUs y más

⚡

Picos (Spike Test)

Aumentos repentinos y drásticos de tráfico. ¿El sistema se recupera después del pico?

Ej: 0 → 500 usuarios en 10 segundos

⏱️

Resistencia (Soak Test)

Carga sostenida durante horas para detectar fugas de memoria o degradación gradual.

Ej: 50 usuarios durante 2 horas

K6: pruebas de carga modernas

# Instalar K6 (Linux/snap)
sudo snap install k6

# lab-04/scripts/carga_basica.js
import http from 'k6/http';
import { sleep, check } from 'k6';

export const options = {
  vus:      50,      // usuarios virtuales
  duration: '30s',  // duración
};

export default function () {
  const res = http.get('http://localhost:8000/items/');

  check(res, {
    'status 200':      r => r.status === 200,
    'tiempo < 500ms':  r => r.timings.duration < 500,
  });

  sleep(1); // pausa entre peticiones
}

# Ejecutar
k6 run scripts/carga_basica.js

Stages, thresholds y reporte

# lab-04/scripts/carga_stages.js
export const options = {
  stages: [
    { duration: '30s', target: 20  }, // rampa subida
    { duration: '1m',  target: 100 }, // carga sostenida
    { duration: '20s', target: 0   }, // rampa bajada
  ],
  thresholds: {
    // El test FALLA si se violan estos límites
    'http_req_duration': ['p(95)<500'],  // 95% bajo 500ms
    'http_req_failed':   ['rate<0.01'],  // menos del 1% errores
    'checks':            ['rate>0.99'],  // +99% checks pasan
  },
};

export default function () {
  const res = http.get('http://localhost:8000/items/');
  check(res, { 'ok': r => r.status < 400 });
  sleep(1);
}

# Generar reporte HTML
K6_WEB_DASHBOARD=true \
K6_WEB_DASHBOARD_EXPORT=reporte.html \
k6 run scripts/carga_stages.js

Métricas clave

Métrica¿Qué mide?
p(50)Mediana de latencia (usuario típico)
p(95)El 95% de usuarios bajo este tiempo
p(99)Casos más lentos — cola larga
http_req_failedTasa de errores HTTP
http_reqsSolicitudes por segundo (RPS)

¿Por qué p(95) y no el promedio? El promedio oculta los picos. El p(95) garantiza que el 95% de usuarios tiene una experiencia aceptable.

Buenas Prácticas

Escribir pruebas de calidad

🎯

Un assert por test

Cada prueba debe fallar por una sola razón. Facilita identificar qué rompió.

📛

Nombres descriptivos

test_crear_usuario_con_email_duplicado_retorna_409 es mejor que test_1.

🔒

Tests independientes

Ningún test debe depender del estado que dejó otro. Usa fixtures para inicializar.

⚡

Pruebas rápidas

Las pruebas lentas no se ejecutan. Unitarias < 100ms, integración < 2s.

🚫

Sin lógica en tests

Evita if, for en tests. Si los necesitas, extrae una función auxiliar.

🔄

Tests antes del fix

Ante un bug, primero escribe el test que lo reproduce, luego corrígelo (TDD reactivo).
