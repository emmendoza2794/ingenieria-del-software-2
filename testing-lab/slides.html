Pruebas Unitarias

Probar funciones puras de forma completamente aislada.
17 tests · cobertura 92%

¿Qué son?

Pruebas Unitarias

🔬

Definición

Prueban una sola unidad de código (función, método, clase) en completo aislamiento. Sin base de datos, sin red, sin archivos externos. Si una dependencia externa es necesaria, se simula (mock).

Cuándo usarlas

  • Lógica de negocio: cálculos, validaciones, reglas
  • Algoritmos: hashear contraseñas, generar tokens JWT
  • Transformaciones de datos: parseos, formateos
  • Casos borde: contraseña vacía, email inválido, token expirado

Cuándo NO usarlas

  • Para verificar que un endpoint HTTP devuelve 200
  • Para verificar que algo se guardó en la base de datos
  • Para verificar que dos módulos se comunican bien

Eso es trabajo de las pruebas de integración.

Las tres propiedades clave:

Rápidas — milisegundos por test, cientos en segundos.
Aisladas — sin dependencias externas reales.
Deterministas — mismo input → mismo resultado siempre.

        {"sub": "user@test.com"},
        expires_delta=timedelta(minutes=1)
    )

¿Qué son?

Pruebas de Integración

🔗

Definición

Prueban que varios módulos trabajan correctamente juntos. Incluyen BD real (en memoria), rutas HTTP, repositorios y servicios reales, sin simularlos con mocks.

Cuándo usarlas

  • Verificar que un endpoint devuelve el código correcto
  • Verificar que los datos se guardan realmente en BD
  • Probar flujos completos: registro → login → token
  • Probar comportamientos con estado: bloqueos, contadores

Diferencia con pruebas unitarias

UnitariaIntegración
Base de datosNoSí (en memoria)
HTTP realNoSí (TestClient)
Velocidadms~1 s por test
Qué detectaBug en lógica puraContrato entre capas roto

Una prueba unitaria que pasa no garantiza que el endpoint funcione. Necesitas integración para cerrar esa brecha.

pytest · fixtures

¿Qué es conftest.py?

⚙️

Archivo especial de pytest

pytest lo carga automáticamente antes de cualquier test del mismo directorio. Aquí se definen fixtures: funciones que preparan el estado antes del test y lo limpian después.

Ciclo de vida de un fixture

1. pytest detecta que el test pide el fixture por nombre de parámetro
↓
2. Ejecuta el fixture hasta el yield → setup (crear BD, cliente HTTP)
↓
3. Corre el test usando lo que devolvió yield
↓
4. Ejecuta lo que viene después del yield → teardown (borrar tablas)

Alcance (scope) del fixture

ScopeSe crea/destruyeCuándo usarlo
functionPor cada testBD limpia por test ✓
classPor clase de testTests relacionados
modulePor archivoSetup costoso
sessionUna sola vezConexión persistente

En nuestro lab: usamos scope="function" (el default) para que cada test de integración tenga su propia BD en memoria desde cero. Así ningún test contamina al siguiente.

FastAPI TestClient + SQLite

El conftest.py que usamos

# tests/integration/conftest.py
from sqlalchemy import create_engine
from sqlalchemy.pool import StaticPool

# StaticPool: todas las conexiones comparten
# la misma BD en memoria (necesario para SQLite)
engine = create_engine(
    "sqlite:///:memory:",
    connect_args={"check_same_thread": False},
    poolclass=StaticPool,
)

@pytest.fixture(scope="function")
def db_session():
    Base.metadata.create_all(engine)
    session = SessionLocal()
    yield session
    session.close()
    Base.metadata.drop_all(engine)

@pytest.fixture
def client(db_session):
    # Resetea el rate limiter entre tests
    try: limiter._storage.reset()
    except: pass

    def override_get_db(): yield db_session
    app.dependency_overrides[get_db] = override_get_db
    # raise_server_exceptions=False → errores devuelven 5xx
    with TestClient(app, raise_server_exceptions=False) as c:
        yield c
    app.dependency_overrides.clear()
# tests/integration/test_auth.py (extracto)

class TestLogin:
    def test_login_exitoso_devuelve_token(
        self, client
    ):
        register(client)
        res = login(client)
        assert res.status_code == 200
        body = res.json()
        assert "access_token" in body
        assert body["token_type"] == "bearer"

    def test_login_bloqueo_progresivo(
        self, client, db_session
    ):
        register(client)
        for _ in range(3):
            login(client, password="wrong")
        # 4.° intento mientras bloqueado → extiende a 2 min
        res = login(client, password="wrong")
        assert res.status_code == 429
        user = db_session.query(User).first()
        remaining = (user.locked_until - datetime.utcnow()).total_seconds()
        assert 100 < remaining <= 120  # ~2 minutos

Simular un usuario real en el navegador.
JavaScript · Nuxt 3 · 5 tests · headless y headed

¿Qué son?

Pruebas End-to-End (E2E)

🌐

Definición

Simulan a un usuario real usando la aplicación completa: browser real, frontend, backend y base de datos, sin ningún mock. Prueban el sistema de punta a punta.

Cuándo usarlas

  • Flujos críticos: login, registro, checkout
  • Smoke test antes de subir a producción
  • Verificar que el front y back se conectan
  • Probar navegación y redirecciones

Lo que verifican (ejemplos reales)

  • ¿Puede el usuario iniciar sesión con credenciales válidas?
  • ¿Aparece el mensaje de error con credenciales incorrectas?
  • ¿La URL cambia a / después de un login exitoso?
  • ¿El formulario de registro tiene todos sus campos?

La trampa: las E2E son lentas (~5 s cada una), frágiles (un cambio de ID rompe el test) y difíciles de depurar.

No reemplaces unitarias ni integración con E2E. Úsalas solo para el camino feliz más importante.

Playwright · JavaScript

Tests E2E contra Nuxt 3

// tests/e2e/login.spec.js
import { test, expect } from '@playwright/test'
import { apiRegister, goToLogin } from './helpers.js'

let testUser

// Registrar 1 usuario antes de todos los tests
test.beforeAll(async () => {
  testUser = { email: `e2e_${suffix()}@test.com`,
               password: 'Test1234!', name: 'E2E User' }
  await apiRegister(testUser.email, testUser.password, testUser.name)
})

test('login exitoso redirige a /', async ({ page }) => {
  await goToLogin(page)                          // goto + waitForHydration
  await page.fill('#login-email', testUser.email)
  await page.fill('#login-password', testUser.password)
  await page.click('#btn-login')
  await page.waitForURL('http://localhost:3000/')
  expect(page.url()).toBe('http://localhost:3000/')
})

test('login fallido muestra mensaje de error', async ({ page }) => {
  await goToLogin(page)
  await page.fill('#login-email', testUser.email)
  await page.fill('#login-password', 'wrong')
  await page.click('#btn-login')
  await expect(page.locator('#login-error')).toBeVisible()
})
// tests/e2e/helpers.js
const API_URL = 'http://localhost:8000'

export async function apiRegister(email, password, name) {
  const body = new URLSearchParams({ email, password, name })
  const res = await fetch(`${API_URL}/auth/register`, {
    method: 'POST',
    headers: { 'Content-Type': 'application/x-www-form-urlencoded' },
    body: body.toString(),
  })
  // 201 creado, 400 ya existe → ambos OK para setup
  if (res.status !== 201 && res.status !== 400)
    throw new Error(`apiRegister falló: ${res.status}`)
}

export async function goToLogin(page) {
  await page.goto('/login', { waitUntil: 'load' })
  // Espera hidratación de Vue 3 (SSR → client handoff)
  await page.waitForTimeout(1000)
  await page.waitForSelector('#btn-login', { state: 'visible' })
}

Truco clave: Nuxt 3 hace SSR antes de que Vue hidrate. El waitForTimeout(1000) da tiempo a que los handlers @click se conecten.

Pruebas de Carga con K6

Medir cuánto aguanta el sistema antes de degradarse.
ISO/IEC 25010 · smoke · load · stress

¿Qué son?

Pruebas de Carga

📈

Definición

No prueban si algo funciona, sino si aguanta. Miden el rendimiento del sistema bajo múltiples usuarios simultáneos: latencia, throughput y tasa de error.

Métricas que miden

  • p95 latencia — el 95% de requests responde en menos de X ms
  • Tasa de error — % de requests que fallan (5xx)
  • Throughput — requests por segundo sostenibles
  • VU (Virtual Users) — usuarios concurrentes simulados

Cuándo usarlas

  • Antes de un lanzamiento público
  • Después de un cambio de arquitectura
  • Cuando hay SLAs de rendimiento definidos
  • Para encontrar cuellos de botella

ISO/IEC 25010 — Eficiencia de Desempeño:

Comportamiento temporal — tiempo de respuesta, throughput.
Utilización de recursos — CPU, memoria, errores bajo carga.
Capacidad — límite máximo de usuarios concurrentes.

K6 · tres tipos

¿Cuándo usar Smoke, Load o Stress?

🕯️

Smoke Test

1 VU · 30 segundos

Pregunta: ¿el sistema arranca y responde algo razonablemente?

Es el mínimo viable. Se corre primero para verificar que el entorno está bien configurado. Si falla, no tiene sentido seguir.

Si el smoke falla → no corras load ni stress.
⚖️

Load Test

10 VUs · ramp-up → sostenido → ramp-down

Pregunta: ¿el sistema cumple los SLAs con la carga normal esperada?

Simula el tráfico típico de producción. Mide si latencia y tasa de error se mantienen dentro de los umbrales aceptables.

El más relevante para decisiones de producción.
💥

Stress Test

5 → 15 → 30 → 50 → 70 VUs

Pregunta: ¿cuál es el punto de quiebre? ¿qué componente cede primero?

Aumenta la carga progresivamente hasta que el sistema falla o se degrada. Se espera que falle — eso es exactamente lo que se busca encontrar.

Esperas que falle — eso es lo que buscas.
K6 · tres niveles

Smoke → Load → Stress

// tests/load/smoke.js — sanity check mínimo
export const options = {
  vus: 1, duration: '30s',
  thresholds: {
    http_req_failed:   ['rate<0.01'],  // < 1% errores
    http_req_duration: ['p(95)<500'],  // 95% < 500ms
  },
}

// tests/load/load.js — carga normal (ISO 25010)
export const options = {
  stages: [
    { duration: '30s', target: 10 },  // ramp-up
    { duration: '90s', target: 10 },  // sostenido
    { duration: '30s', target: 0  },  // ramp-down
  ],
  thresholds: {
    http_req_duration: ['p(95)<800'],   // temporal
    http_req_failed:   ['rate<0.02'],   // utilización
    login_duration:    ['p(95)<600'],
  },
}

// tests/load/stress.js — punto de quiebre
// 5 → 15 → 30 → 50 → 70 → 0 VUs (30s cada etapa)
export const options = {
  stages: [
    { duration: '30s', target: 5  },
    { duration: '30s', target: 15 },
    { duration: '30s', target: 30 },
    { duration: '30s', target: 50 },
    { duration: '30s', target: 70 },
    { duration: '30s', target: 0  },
  ],
}

Ejecutar desde MVP/back/

# Sanity check básico
k6 run tests/load/smoke.js

# Carga normal de producción
k6 run tests/load/load.js

# Buscar el punto de quiebre
k6 run tests/load/stress.js

Flujo que ejercitan

setup() registra el usuario una vez.
Cada VU: POST /auth/login → obtiene token → GET /auth/me

ISO/IEC 25010: el load test mide eficiencia de desempeño — comportamiento temporal (latencia) y utilización de recursos (tasa de error).

Buenas Prácticas

Lecciones aprendidas en el lab

Principios — aplican a cualquier tipo de prueba

🎯

Un assert por test

Cada prueba falla por una sola razón. Sabes exactamente qué se rompió sin tener que leer tres mensajes de error.

📛

Nombres que describen el comportamiento

test_login_bloqueo_progresivo_intento_4_bloquea_2_minutos
El nombre es la especificación.

🔗

Tests independientes entre sí

Un test no puede depender del estado que dejó otro. Cada uno debe poder correr solo, en cualquier orden.

Técnicas específicas — por tipo de prueba

Integración

  • → StaticPool + drop_all — BD limpia en cada test
  • → limiter._storage.reset() — rate limiter sin acumulación entre tests

Unitarias

  • → freezegun — congela el reloj del sistema para probar tokens expirados sin esperar realmente

E2E — Playwright + Nuxt 3

  • → waitForTimeout(1000) — da tiempo a Vue para conectar sus handlers después de SSR
  • → workers: 1 — un test a la vez evita saturar el rate limiter del back
Resumen

Las pruebas no son opcionales,
son ingeniería.

Las unitarias prueban la lógica pura sin nada externo.
Las de integración prueban que los módulos trabajan juntos.
Las E2E prueban que el usuario puede cumplir su objetivo.
Las de carga prueban que el sistema aguanta en producción.
